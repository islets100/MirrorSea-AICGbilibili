### 1. MinIO 与 XXL-JOB 是什么？为什么用在【大文件断点续传】？
*(对应简历第一点)*

*   **这是什么？**
    *   **MinIO**：你可以把它理解为一个**开源的、免费的“阿里云 OSS”或“百度云保存网盘”**。因为视频文件很大，存 MySQL 肯定不行，存普通服务器的本地文件夹也不方便集群共享，所以用 MinIO 这个专门的对象存储服务器来存文件。
    *   **XXL-JOB**：这是一个分布式定时任务调度框架。你可以把他当你手机上的“闹钟”。为什么不用 SpringBoot 自带的 `@Scheduled`？因为如果有 3 台服务器，自带的定时器会同时响 3 次导致重复执行，而 XXL-JOB 有个调度中心，保证时间到了只让一台机器去干活。
*   **源码对应在哪？**
    *   在 `MinioServiceImpl.java` 里。对于大文件，前端把它切成了 100 块发给后端存进 MinIO。合并的时候，源码调用了 **`composeObject`** 接口。
*   **面试你怎么讲？**
    *   你就说：“如果把 100 个 10MB 的分片拉回 Java 内存里用流 `InputStream` 拼接，这 1GB 会直接把 JVM 内存撑爆（OOM）。我调用 MinIO 的 `composeObject` 接口，这其实是发送了一条指令给存储服务器，让它自己在底层硬盘上把块合并了，这就叫**零拷贝（Zero-Copy）接合**，极大降低了 Java 后端的压力。对于残缺的碎片，我配置了 **XXL-JOB 定时任务**，每天半夜去 MinIO 里把超出 24 小时没合并的垃圾分片删掉。”

### 2. Redis List 缓冲与 ES的异构同步（宽门架构）是什么？
*(对应简历第二点)*

*   **这是什么？**
    *   你了解 RocketMQ，本来业务插完 MySQL，发个 MQ，消费者收到 MQ 直接连 ES 调用 `client.bulk()` 就行了。但高并发下，每一条弹幕/点赞都去更新 ES，大量零碎的网络请求会让 ES 随时宕机。这就需要个“蓄水池”。
*   **源码对应在哪？**
    *   在 `MysqlToEsConsumer.java` 中，消费者收到 MQ 后**没有直接写 ES**，而是调用 Redis 往下插数据：`objectRedisTemplate.opsForList().rightPush(VIDEO_ADD_KEY, map)`。利用 **Redis List（先进先出队列）** 做了个缓冲池。
    *   然后在 `MysqlToEsHandler.java` 里（这也是个 XXL-JOB 定时任务），每隔 1 分钟，把 Redis 里攒的几千条数据一口气掏出来，打包成**一个 ES 的 BulkRequest** 发给 ES。
*   **面试你怎么讲？**
    *   你就说：“单兵作战极其损耗 ES 的 Segment 倒排索引合并性能。我用 Redis List 做缓冲，配合定时任务把异步的一万次小网络 I/O 收敛成了单次批处理，保护了 ES，这种设计叫做宽门架构（或者叫大批处理削峰）。”

### 3. Guava 布隆过滤器 (Bloom Filter) 是什么？
*(对应简历第三点)*

*   **这是什么？**
    *   **Guava** 是 Google 提供的一个 Java 工具包。**布隆过滤器**是一种极其省内存的算法，它能**以极其极快的速度判断一个 ID 存不存在**。它的特性是：“它说不存在，那**一定绝对不存在**；他说存在，那**可能存在**”。
*   **面试你怎么讲？**
    *   你就说：“在海量数据往前文提到的 ES 同步时，最怕收到『更新一个根本不存在的视频』的请求。如果直接发给 ES，ES 会抛出 NotFound 的异常风暴，很吃 CPU。我在 Java 内存里加了一层 Guava 的布隆过滤器，每次请求先过一遍，如果布隆过滤器说查无此 ID，我连网络请求都不发，直接在 Java 代码里拦截掉（直接 return）。这能免除 99% 的黑客恶意异常报错。”

### 4. 弹幕点赞的 高频演进与写缓冲架构
*(对应简历第四点)*

*   **这是什么？**
    *   目前源码里 LikeServiceImpl.java 等处理点赞，是直接 `mapper.insert()` 同步写 MySQL 的。
*   **面试你怎么讲（这叫主动抛出痛点来拿分）？**
    *   “我在重构复盘时发现，几十万人同时点赞绝对会让 DB 死锁雪崩。我会这样改造：用户的点赞不连数据库，而是全抛到 **Redis 的 Set 集合** 或者 Hash 中（Redis 在内存里抗并发能力是百万级的）。然后定个闹钟（定时任务），每 10 秒把 Redis 里的最终点赞数，打包变成一条 `update video set like=like+1` 写进 MySQL（这个过程叫定时双写/异步刷盘）。”

### 5. WebSocket 跨服路由与 Redis Pub/Sub（发布/订阅）
*(对应简历第五点)*

*   **这是什么？**
    *   **WebSocket** 是一种让服务器和客户端保持“长连不断开”的技术。你发微信不用每次刷新页面，就是因为有它。源码中聊天服务用的是这个。
    *   **Redis Pub/Sub（广播机制）**：微服务架构（Spring Cloud Alibaba）意味着聊天服务器不止一台（可能会有网关 Gateway 把请求路由到服务器A 和 服务器B）。
*   **面试你怎么讲？**
    *   “现在代码所有的聊天 SessionId 存在自己 JVM 里的 `ConcurrentHashMap` 里。如果在集群模式下，A 和 B 连接在不同的机器上是讲不了话的。遇到这个问题，我在设计时引入了 **Redis 的 Pub/Sub（发布订阅）广播机制**（或者直接用 RocketMQ 的广播模式）。机器 1 收到了发给 B 的消息，查不到 B，就会朝整个 Redis 大喊一声『谁连着 B，帮我把消息给他！』。此时机器 2 听到了广播，查自己的 Map 发现 B 在自己这里，就会代替机器 1 把消息投递给 WebSocket 客户端。”
